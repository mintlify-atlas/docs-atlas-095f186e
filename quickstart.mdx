aube drops into existing Node.js projects without requiring you to change your lockfile format. If your project already has `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, or `bun.lock`, aube reads and writes it in place. This guide walks you from install to daily use in a few minutes.

<Steps>
  <Step title="Install aube">
    The recommended way is via [mise](https://mise.jdx.dev):

    ```bash
    mise use -g aube
    ```

    Verify it's available:

    ```bash
    aube --version
    ```

    For other install methods — Homebrew, npm, cargo, Ubuntu PPA, Fedora COPR, or from source — see the [installation guide](/installation).
  </Step>

  <Step title="Run your first install">
    Navigate to an existing Node.js project and run:

    ```bash
    aube install
    ```

    If the project already has a supported lockfile, aube reads it and writes any updates back to the same file. You do not need to delete or migrate your existing lockfile to try aube. For a new project with no lockfile, aube creates `aube-lock.yaml`.

    <Tip>
      `aube run`, `aube test`, and `aube exec` all check install freshness automatically. If `package.json` or the lockfile changed since the last install, they install first and then run. In day-to-day work you rarely need a separate `aube install` step.
    </Tip>
  </Step>

  <Step title="Add and remove dependencies">
    Add a runtime dependency:

    ```bash
    aube add react
    ```

    Add a dev dependency:

    ```bash
    aube add -D vitest
    ```

    Remove a dependency:

    ```bash
    aube remove react
    ```

    Update all dependencies within their current `package.json` ranges:

    ```bash
    aube update
    ```
  </Step>

  <Step title="Run scripts and binaries">
    Run a script from `package.json`:

    ```bash
    aube run build
    ```

    If the script name does not conflict with an aube command, you can call it directly:

    ```bash
    aube dev
    aube build
    aube lint
    ```

    Run the test script (installs first if anything changed):

    ```bash
    aube test
    ```

    Run a local binary from `node_modules/.bin`:

    ```bash
    aube exec vitest
    ```

    Run a package in a throwaway environment without installing it permanently:

    ```bash
    aube dlx cowsay hi
    ```
  </Step>

  <Step title="Use the aubr and aubx shortcuts">
    `aubr` and `aubx` are multicall shims that ship alongside `aube` in every release. They share a binary with `aube` and dispatch on `argv[0]`, so every flag that works on the full command also works on the shim.

    ```bash
    aubr build          # same as: aube run build
    aubx cowsay hi      # same as: aube dlx cowsay hi
    ```

    No extra setup is needed when you install aube via mise or the tarball — all three binaries are included.
  </Step>

  <Step title="Set up CI">
    Use `aube ci` in CI pipelines where the lockfile must be the source of truth. It removes `node_modules`, verifies the lockfile is fresh for the current `package.json`, then installs:

    ```bash
    aube ci
    ```

    For Docker layers or workflows where you only want to update the lockfile without touching `node_modules`:

    ```bash
    aube install --lockfile-only
    ```

    For production-only installs that skip `devDependencies`:

    ```bash
    aube install --prod
    ```

    <Tip>
      `aube ci` is stricter than `aube install`. It fails if the lockfile is out of sync with `package.json`, which catches accidental lockfile drift in pull requests.
    </Tip>
  </Step>

  <Step title="Try workspace commands">
    aube supports workspace projects and the `workspace:` protocol. If your project already uses `pnpm-workspace.yaml`, aube can read and write it. New aube-first workspaces use `aube-workspace.yaml`.

    Install dependencies across all workspace packages:

    ```bash
    aube install -r
    ```

    Run the test script in every package:

    ```bash
    aube run test -r
    ```

    Add a dependency to a specific workspace package:

    ```bash
    aube add zod --filter @acme/api
    ```

    For a full walkthrough of filters, catalogs, and workspace dependencies, see the [workspaces guide](/guides/workspaces).
  </Step>
</Steps>
